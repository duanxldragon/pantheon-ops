# 动态菜单能力成熟度评估与演进蓝图（2026-04-22）

更新时间：2026-04-22

类型：Assessment
归属层：platform
状态：Active

## 1. 评估范围

本次评估聚焦三个层面：

- `platform`：动态菜单装配、导航策略、模块注册方式。
- `system/iam`：菜单、页面、按钮、接口四层权限与授权链路。
- `business/cmdb`：作为第一个业务域样例，验证新增业务模块是否真正符合“灵活接入”的初始目标。

本文件不把 `auth / iam / org / config` 混为一个模块；这里只讨论“动态菜单能力”本身，以及它对业务模块扩展的影响。

## 2. 初始目标回顾

根据 `DESIGN.md` 与模块契约，Pantheon Base 对动态菜单的原始诉求不是“做一个能显示树形菜单的后台”，而是：

- 菜单作为导航元数据独立建模；
- 菜单与页面、按钮、接口权限解耦；
- 业务模块通过注册方式挂接到底座，而不是在 Layout 里硬编码；
- 新增业务时，菜单、权限、i18n、审计能一起闭环。

这意味着动态菜单的核心目标有两个：

1. **导航灵活性**
2. **业务接入一致性**

而不是一步到位追求“只改数据库就能上线新页面”。

## 3. 当前成熟度结论

结论：Pantheon Base 当前处于 **L3「注册式动态菜单」** 阶段，尚未进入 **L4「受控组件注册表驱动」**。

### 3.1 成熟度分级

| 等级 | 名称 | 特征 | Pantheon 当前状态 |
| :--- | :--- | :--- | :--- |
| `L0` | 静态菜单 | Layout 写死菜单与路由 | 已超过 |
| `L1` | 动态菜单树 | 后端返回菜单树，前端按权限渲染 | 已达到 |
| `L2` | 权限解耦菜单 | 菜单/页面/按钮/接口分层授权 | 已达到 |
| `L3` | 注册式动态菜单 | 后端管导航元数据，前端通过 manifest 注册页面组件 | **当前所在阶段** |
| `L4` | 受控组件注册表 | 后端 `component` 通过前端白名单注册表解析，无需每次改装配入口 | 待实现 |
| `L5` | 插件化业务装配 | 业务模块具备更强的自动发现、能力声明与发布治理 | 长期目标 |

### 3.2 为什么是 L3

当前已经具备：

- 后端 `system_menu` 管理菜单树、`title_key`、`component`、`route_name`、`module` 等元数据；
- 导航树按角色菜单授权裁剪，不允许前端拿全量后自行隐藏；
- 页面权限与按钮权限已从菜单可见性中拆开；
- 业务模块可以通过 `ModuleConfig` 统一声明 `routes / menus / permissions / i18nNamespaces`；
- `business/cmdb` 已作为样例接入，验证了菜单 seed、权限 seed、详情页高亮和导入导出闭环。

但当前仍然需要：

- 新增前端页面组件文件；
- 新增业务模块 `index.ts` manifest；
- 在 `frontend/src/core/router/modules.ts` 中显式注册模块。

这说明当前系统实现的是：

> **后端动态菜单 + 前端显式模块注册**

而不是：

> **后端菜单配置即可独立上线新业务页面**

## 4. 当前能力矩阵

### 4.1 已达成能力

| 能力项 | 当前状态 | 说明 |
| :--- | :--- | :--- |
| 导航树动态返回 | 已达成 | 由后端菜单树与角色授权决定可见导航 |
| 菜单标题国际化 | 已达成 | 使用 `title_key`，前端翻译 |
| 菜单与权限解耦 | 已达成 | 导航、页面、按钮、接口分层 |
| 业务模块接入闭环 | 已达成 | CMDB 已覆盖菜单、权限、i18n、审计 |
| 详情页菜单高亮 | 已达成 | 支持 `activeMenu` |
| 动态菜单管理页 | 已达成 | 可维护 routeName、module、pagePerm 等 |
| 后端零代码新增页面 | 未达成 | 仍依赖前端页面组件与 manifest |
| 组件注册表驱动装配 | 未达成 | 还没有统一 `component -> lazy component` 白名单映射 |
| manifest 与菜单 seed 一致性校验 | 部分达成 | 有契约，但缺少自动比对与发布校验 |

### 4.2 企业级视角评分

| 维度 | 评分 | 判断 |
| :--- | :--- | :--- |
| 导航灵活性 | `8/10` | 已满足大多数后台导航调整诉求 |
| 权限一致性 | `8.5/10` | 已形成企业级后台应有的分层授权 |
| 业务模块接入效率 | `7/10` | 已可复制扩展，但还需要显式注册 |
| 运行时可控性 | `8/10` | 当前模式比完全自由动态加载更安全 |
| 插件化程度 | `5.5/10` | 还停留在 manifest 注册阶段 |

综合判断：**当前实现符合企业级后台的“稳态动态菜单”，但还不是高插件化业务平台。**

## 5. CMDB 接入评估

`business/cmdb` 已经成为动态菜单能力的第一块真实验证样板。

### 5.1 已完成的接入闭环

- 后端模块已独立装配：`backend/modules/business/cmdb/module.go`
- 后端菜单 seed 已声明 `module / path / component / page_perm / perms / route_name`
- 前端模块 manifest 已独立声明：`frontend/src/modules/business/cmdb/index.ts`
- 前端已包含类型页、实例页、详情页
- 详情页通过 `activeMenu` 维持导航高亮
- 导入导出、错误文件下载、审计参数记录已接入
- 仪表盘已汇总 CMDB 类型数、实例数、活跃实例数

### 5.2 说明了什么

CMDB 证明了两件事：

1. 现在的动态菜单方案已经能支撑真实 `business/*` 模块落地；
2. 但新增业务仍是“**模块注册式接入**”，不是“**纯配置式接入**”。

换句话说，CMDB 已经接入成功，但它也恰好暴露了下一阶段该优化的地方：**页面组件解析仍然收口在前端装配文件，而不是收口为平台级组件注册表。**

## 6. 根因分析

当前系统没有偏离设计，真正的根因是：

### 6.1 当前选择的是“可控灵活”，不是“自由装配”

对于企业级后台，完全让后端菜单数据决定前端任意组件路径，会带来：

- 页面装配不可控；
- 类型系统与编译期校验失效；
- 权限与页面真实能力容易漂移；
- 发布治理和灰度治理难度上升。

因此当前架构故意保留了“前端 manifest 注册”这道门。

### 6.2 现在缺的不是菜单，而是平台级组件注册表

真正的差距不在菜单树，而在于：

- `system_menu.component` 虽然已存在；
- 但前端没有把它提升为受控的 `component key -> lazy component` 注册表；
- 业务模块仍需修改模块装配入口，导致“新增业务灵活性”还没有进一步释放。

## 7. 下一阶段演进蓝图

建议按三个阶段推进，而不是直接跳插件化。

### 7.1 Phase A：收口为“受控组件注册表”

目标：从 `L3` 升级到 `L4`。

核心动作：

- 在 `platform` 层新增统一组件注册表，例如：
  - `business/cmdb/CMDBTypeList`
  - `business/cmdb/CMDBItemList`
  - `business/cmdb/CMDBItemDetail`
- 后端菜单表中的 `component` 只能引用白名单 key；
- 前端路由生成时，优先通过注册表解析页面组件；
- `ModuleConfig.routes` 逐步从“直接持有组件对象”演进成“声明 component key + 页面元数据”；
- 在开发态和 CI 中新增一致性校验：
  - manifest 是否声明了后端 seed 的 path
  - 后端 `component` 是否存在于前端注册表
  - `pagePermission` 是否与菜单 seed 一致

收益：

- 新增页面仍然要交付代码；
- 但可以减少装配散点；
- 业务接入从“改多处”收敛为“注册表 + 模块 manifest + seed 一致性”。

当前首批落地状态：

- `platform` 层已新增前端 `component registry`；
- `business/cmdb` 已成为第一批注册表样板，类型页、实例页、详情页已改为受控 `component key` 解析；
- 菜单管理页已开始对 `business/*` 页面菜单校验已注册组件键。

第二批落地状态：

- `system/*` 页面已逐步收口进同一套 `component registry`；
- `auth` 与 `platform/dashboard` 已补齐到同一套组件键解析链路；
- 前端已新增 `manifest / menu seed` 一致性检查脚本：`cd frontend && npm run check:menu-contract`；
- 当前检查结果已覆盖菜单 path、`titleKey`、`routeName`、`module`、`pagePermission`、`component key` 六类核心契约。

Phase A 收尾状态：

- 后端菜单保存链路已增加 `component key` 白名单校验；
- 菜单管理页已从自由输入升级为注册表自动补全，并继续保留校验；
- `npm run build` 已通过 `prebuild` 自动执行契约检查；
- GitHub Actions 已新增 `Quality Gates`，在 PR 与 `main` 推送时执行菜单契约检查、前端构建与后端测试。

### 7.2 Phase B：增强为“模块级能力声明”

目标：让 `business/*` 更像标准化接入单元。

核心动作：

- 后端模块声明：
  - `ModuleName`
  - `Migrate`
  - `SeedMenus`
  - `SeedPerms`
- 前端模块声明：
  - `scope`
  - `routes`
  - `menus`
  - `permissions`
  - `i18nNamespaces`
  - `componentKeys`
  - `featureFlags`
- 增加模块接入检查脚本，至少校验：
  - 菜单 path 重复
  - 权限 key 冲突
  - i18n namespace 漏注册
  - 详情页 `activeMenu` 缺失

收益：

- 新业务模块接入可模板化；
- AI 与人工都更不容易把 business 能力塞回 system 底座。

### 7.3 Phase C：谨慎探索插件化业务装配

目标：作为长期方向，不作为当前短期必须项。

可考虑的能力：

- 模块包自动发现；
- feature flag 驱动模块启停；
- 业务模块版本与依赖声明；
- 发布前的模块兼容检查。

但当前不建议直接追求：

- 后端随意配置任意组件路径；
- 运行时远程加载不受控页面；
- 没有注册表白名单的动态 import。

## 8. 面向 CMDB 的具体集成蓝图

CMDB 应作为 Phase A 的第一批验证对象。

### 8.1 平台层动作

- 为 CMDB 三个页面建立标准 `component key`
- 把当前后端 seed 中的 `component` 视为正式契约字段
- 在菜单管理页把 `component` 明确展示为“注册表组件键”，避免被误解为任意路径

### 8.2 CMDB 模块动作

- 保持 `business.cmdb` 作为业务域归属，不回流到 `system/*`
- 后续新增 CMDB 详情关系视图、拓扑摘要、批量导入审计页时，继续沿用：
  - 模块独立 manifest
  - 菜单 seed
  - 页面权限
  - 操作权限
  - i18n namespace
  - 审计点
- 若新增“CMDB 工作台”或跨类型概览，应归到 `platform` 聚合层，不要塞进 `system/config`

### 8.3 建议优先级

CMDB 下一批最值得先做的，不是再加更多菜单层级，而是：

1. 详情页关系视图标准化
2. 导入失败明细与审计追踪增强
3. component key 与页面注册表对齐

这样能同时推进业务能力与平台装配能力，而不是只堆业务页面。

## 9. 验收标准

当以下条件成立时，可认为动态菜单进入下一成熟阶段：

- 新页面不再需要修改 `frontend/src/core/router/modules.ts`
- 后端 `system_menu.component` 全部命中前端注册表白名单
- manifest、menu seed、permission seed 可做自动一致性检查
- 新增 `business/*` 模块时，仍然保持菜单、权限、i18n、审计闭环
- 未授权角色无法看到导航，也无法绕过页面守卫进入业务页

当前 Phase A 已基本满足前四项；后续新增第二个 `business/*` 模块时，应继续用相同检查链路验证重复性。

## 10. Pantheon 专家结论

Pantheon Base 当前的动态菜单实现**没有脱离最初预期**，而且已经达到企业级后台比较合理的稳态：

- 它不是简单的静态菜单；
- 它也不是危险的任意动态装配；
- 它是一个已经能承载 `business/cmdb` 落地的 **L3 注册式动态菜单平台**。

下一阶段不应推翻现有设计，而应继续沿着下面这条线演进：

> **后端继续持有导航元数据，前端把页面装配进一步收口为受控组件注册表。**

这条路线最符合 Pantheon Base 的平台化方向，也最适合把 CMDB 作为第一个业务域样板持续演进。
