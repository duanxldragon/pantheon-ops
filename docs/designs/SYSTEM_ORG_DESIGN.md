# system/org 组织域设计

更新时间：2026-05-05

类型：Design
归属层：system/org
状态：Active

本文补齐 `system/org` 的独立设计锚点。组织域当前已有部门、岗位、组织治理页面和后端能力，但历史上主要散落在总体设计、系统审计和页面实现中，缺少单域设计文档。

---

## 1. 归属与边界

`system/org` 负责组织结构和岗位治理。

覆盖：

- 部门树。
- 岗位。
- 部门负责人。
- 用户的组织归属。
- 组织健康与治理任务。

不覆盖：

- 登录、会话、MFA、安全中心，归属 `system/auth`。
- 用户 CRUD、角色、菜单、权限工作台，归属 `system/iam`。
- 字典、设置、i18n、上传、生成器，归属 `system/config`。
- 租户。组织不是租户，部门不能被当成 tenant 使用。

## 2. 核心对象

| 对象 | 表 | 说明 |
| :--- | :--- | :--- |
| Dept | `system_dept` | 部门树和组织层级 |
| Post | `system_post` | 岗位，必须归属具体部门 |
| User Org Binding | `system_user.dept_id/post_id` | 用户组织归属 |

## 3. 部门治理

部门能力必须满足：

- 根节点不可删除、不可移动到其他父节点。
- 删除部门前必须清理子部门、岗位和用户绑定。
- 部门负责人优先绑定真实用户，而不是长期保留自由文本。
- 支持治理任务识别：无负责人、无岗位、空部门。

## 4. 岗位治理

岗位能力必须满足：

- 岗位必须归属具体部门，不能直接挂到根节点。
- 用户选择岗位时，岗位必须与用户部门一致。
- 岗位禁用后，应明确是否阻止新增用户绑定。

## 5. 与 iam 的边界

- `system/org` 维护组织结构。
- `system/iam` 维护用户、角色和权限。
- 用户页面可以展示和编辑部门/岗位归属，但组织规则仍由 `system/org` 解释。
- 数据权限未来可以消费组织结构，但不能把数据权限规则写回组织模块内部。

## 6. 数据权限扩展位

组织域是未来数据权限的关键依赖，但不是数据权限引擎。

当前约束：

- `DataScopeReq.DeptID` 表示当前主体部门。
- `DataScopeReq.DeptIDs` 表示自定义部门集合。
- `WithDataScope` 使用 `dept_id` 注入行级过滤。

后续若要支持“本部门及以下”，必须明确部门树展开逻辑来自 `system/org`，SQL 注入仍由 `pkg/database` 或数据权限中间件承载。

## 7. 验收

固定验收页面：

- `/system/dept`
- `/system/post`
- `/system/user` 中部门/岗位选择链路

固定验收项：

- 部门树加载、创建、编辑、删除。
- 部门负责人候选人约束。
- 岗位创建、编辑、删除。
- 用户部门与岗位一致性。
- 组织治理任务导出。
- 权限不足时页面或按钮不可用。
- i18n key 完整。
- 关键写操作审计可追踪。
