# 运维域业务错误语义附录  English version: [BUSINESS_ERROR_SEMANTICS_APPENDIX.en.md](./BUSINESS_ERROR_SEMANTICS_APPENDIX.en.md)  更新时间：2026-05-11  类型：Design 归属层：business/* 状态：Active  本文是 `pantheon-ops` 业务模块在真实实现阶段的**错误 key canonical 附录**。  目标只有一个：  - 让后端返回的业务错误 key 在编码前就收口 - 让前端 i18n、接口联调、验收、审计详情查看同一套语义 - 避免每个 handler / service 在实现时各自发明一套错误命名  适用范围：  - `business/cmdb` - `business/deploy` - 后续新增的 `business/*` 模块  ---  ## 1. 规则  ### 1.1 canonical 命名  `pantheon-ops` 业务错误 key 统一使用：  ```text business.<module>.<resource>.<reason> ```  示例：  - `business.cmdb.host.notFound` - `business.cmdb.collect.sshAuthFailed` - `business.deploy.task.invalidStartState` - `business.deploy.taskHost.markFailed.reasonRequired`  约束：  - `module` 取业务模块名，如 `cmdb`、`deploy` - `resource` 取资源名或明确交互域，如 `host`、`group`、`label`、`collect`、`package`、`task`、`taskHost` - `reason` 使用 lowerCamelCase - 禁止继续新增历史短 key，如 `cmdbhost.*`、`cmdbgroup.*`、`cmdblabel.*`  ### 1.2 返回责任  - 后端负责返回**稳定错误 key** - 前端负责把错误 key 翻译为本地化文案 - 审计详情、导入导出结果、页面内 banner / toast 统一优先按错误 key 翻译，而不是硬编码自然语言  ### 1.3 HTTP 语义建议  | 场景 | 建议 HTTP | 错误 key 语义 | | :--- | :--- | :--- | | 参数缺失 / 格式错误 | `400` | `*.nameRequired`、`*.invalidLabel` | | 权限不足 | `403` | `permission.denied` 或业务明确 `*.forbidden` | | 资源不存在 | `404` | `*.notFound` | | 唯一键冲突 / 状态冲突 | `409` | `*.ipExists`、`*.nameVersionConflict`、`*.invalidStartState` | | 外部依赖失败 / 执行失败 | `422` 或 `500` | `*.sshAuthFailed`、`*.executionFailed` |  说明：  - 平台通用错误继续复用 base 的 `param.invalid`、`permission.denied`、`request.failed` 等 - 只有进入明确业务语义时，才返回 `business.*`  ---  ## 2.
