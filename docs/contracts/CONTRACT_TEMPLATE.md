# 标题

一句话说明这份合同要锁定的边界。

---

## 1. 背景

- 为什么需要这份合同
- 当前混乱、风险或漂移点是什么
- 为什么不能只靠单个设计文档解决

## 2. 归属层

- 明确属于 `platform`、`system/auth`、`system/iam`、`system/org`、`system/config` 或 `business/*`
- 如果跨层，明确主归属层和从属依赖

## 3. 目标

- 本合同要达成什么
- 达成后项目会获得什么稳定性或一致性

## 4. 非目标

- 本轮明确不解决什么
- 哪些能力只是预留，不进入当前完成范围

## 5. 边界

覆盖对象：

- 页面
- 模块
- API
- 数据结构
- 流程

不覆盖对象：

- `TBD`

## 6. 依赖

- 依赖的设计文档
- 依赖的公共组件或公共流程
- 依赖的数据库或模块契约

## 7. 强约束

- 后续设计和实现必须遵守的红线
- 允许依赖什么
- 禁止依赖什么
- 禁止出现什么回归

## 8. 完成定义

至少从以下角度描述：

- 功能边界
- 视觉或交互边界
- 权限边界
- i18n 边界
- 审计或安全边界
- 文档与验收闭环边界

- 清单
- 模板
- 验收矩阵
- 构建命令
- 测试命令
- 扫描规则

## 11. 状态维护规则

- 当新的设计或整改不再满足本合同时，必须先更新合同，再改实现
- 当本合同被新的合同覆盖时，应把当前文档改为 `状态：Superseded`
- 当本合同退出当前治理主链但仍需保留参考价值时，应改为 `状态：Archived`
